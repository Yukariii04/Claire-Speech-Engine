# 1. Executive Summary

The Performance Representation defines the official backend-independent description of speech performance within the Claire Project.

It represents **how speech should be performed** rather than **how speech should be generated**.

Every implementation of the Claire Performance Engine SHALL produce a valid Performance Representation.

Every compatible backend SHALL interpret the Performance Representation according to its supported capabilities.

The Claire Acoustic Model SHALL use the complete Performance Representation as its native input.

-------------------------------------------------------------------------------

# 2. Background

Traditional speech synthesis systems combine communication understanding with acoustic generation.

This tightly couples expressive behavior to a specific acoustic model.

The Claire Project separates these concerns through a backend-independent Performance Representation.

This representation becomes the common language shared between communication reasoning and speech rendering.

-------------------------------------------------------------------------------

# 3. Problem Statement

Without a common representation:

- Communication reasoning cannot evolve independently.
- Acoustic models become tightly coupled to expressive logic.
- Backend switching changes behavior.
- Research cannot be shared between synthesis engines.
- Future acoustic models require redesign.

The Performance Representation eliminates these limitations.

-------------------------------------------------------------------------------

# 4. Objectives

The Performance Representation SHALL:

- Represent ideal speech performance.
- Remain completely backend-independent.
- Remain immutable once generated.
- Support multiple compatible backends.
- Serve as the native interface for CAM.
- Allow future extension without breaking compatibility.

-------------------------------------------------------------------------------

# 5. Scope

## In Scope

- Performance description.
- Backend-independent information.
- Communication output.
- Stable representation.
- Versioning.
- Future extensibility.

## Out of Scope

- Waveforms.
- Phonemes.
- Spectrograms.
- Backend implementation.
- Acoustic inference.
- Neural network architecture.
- Runtime implementation.

-------------------------------------------------------------------------------

# 6. Design Principles

The Performance Representation SHALL satisfy the following principles.

### Backend Independence

No field may depend upon a specific acoustic model.

---

### Immutability

Once generated by CPE, the Performance Representation SHALL NOT be modified.

---

### Extensibility

Future versions may introduce additional fields without invalidating existing representations.

---

### Compatibility

Older Performance Representations SHALL remain readable whenever practical.

---

### Separation of Responsibility

The Performance Representation describes performance.

It does not describe synthesis.

-------------------------------------------------------------------------------

# 7. Performance Context

The Performance Representation is generated from a Performance Context.

Current inputs are:

Required

- Text

Optional

- Character State

Future versions may extend Performance Context while maintaining backward compatibility.

-------------------------------------------------------------------------------

# 8. Representation Requirements

Every valid Performance Representation SHALL:

- Preserve the original text.
- Remain backend-independent.
- Describe intended speech performance.
- Avoid implementation details.
- Remain deterministic for identical inputs under identical conditions.

-------------------------------------------------------------------------------

# 9. Prohibited Information

The Performance Representation SHALL NOT contain:

- Backend-specific commands.
- Acoustic parameters.
- Spectrogram data.
- Waveform information.
- Neural network tensors.
- GPU-specific information.
- Runtime implementation details.

These belong exclusively to renderers.

-------------------------------------------------------------------------------

# 10. Compatible Backend Responsibilities

Compatible backends SHALL:

- Accept a valid Performance Representation.
- Interpret supported information.
- Ignore unsupported information safely.
- Never invalidate the representation.

Capability degradation is considered expected behavior.

-------------------------------------------------------------------------------

# 11. Claire Acoustic Model Responsibilities

The Claire Acoustic Model SHALL:

- Accept the complete Performance Representation.
- Interpret every standardized field.
- Produce speech with minimal information loss.
- Remain compatible with future representation versions whenever practical.

-------------------------------------------------------------------------------

# 12. Versioning

Every Performance Representation SHALL contain a version identifier.

Future versions SHALL extend existing representations rather than redefine previously standardized fields whenever practical.

-------------------------------------------------------------------------------

# 13. Future Extensions

Future versions may introduce:

- Additional contextual information.
- New expressive attributes.
- Additional performance metadata.
- Enhanced compatibility mechanisms.

These extensions SHALL preserve the architectural principles established by this specification.

-------------------------------------------------------------------------------

# 14. Success Criteria

This specification is considered complete when:

- Performance Representation is clearly defined.
- Responsibilities are separated.
- Backend independence is preserved.
- Future extensibility is documented.
- CAM integration is supported.

-------------------------------------------------------------------------------

# 15. AI Instructions

When extending the Performance Representation:

1. Preserve backend independence.
2. Preserve immutability.
3. Preserve compatibility.
4. Avoid renderer-specific information.
5. Maintain Constitution compliance.
6. Maintain Lexicon terminology.
7. Preserve CPE as the sole producer of the representation.
