# Why this PRD matters

This is the first PRD where someone outside the project can use CSE without knowing anything about its internals.

From this point on, the public API becomes a contract. We should be careful about changing it because other projects may eventually depend on it.

---

# One architectural improvement

Now that we're exposing a public API, I want to establish an internal rule:

```
src/cse/
│
├── api/        ← Public, stable surface
├── core/       ← Internal implementation
├── language/
├── performance/
├── runtime/
├── streaming/
├── voice/
└── backends/
```

**Nothing outside `api/` should be considered part of the public API.**

That gives us freedom to refactor internals later without breaking users.

---

# Repository maturity after PRD-009

At this point, CSE becomes:

- ✅ A library
- ✅ An engine
- ✅ A documented API
- ✅ Backend-agnostic
- ✅ Ready for packaging

We are no longer building "project code." We're building something that other developers can import into their own applications.

## One recommendation before you implement

This is the point where we should start thinking like maintainers of a public library.

I recommend adopting a simple rule:

> **Anything imported from `cse` is public and should remain stable. Everything else is internal and may change between releases.**

For example:

```python
from cse import SpeechEngine
```

should be considered part of the long-term API.

On the other hand:

```python
from cse.performance.compiler import PerformanceCompiler
```

should be treated as an internal implementation detail unless you explicitly decide to make it public in the future.

That distinction will make future refactoring much easier while giving users confidence that the main API won't break unexpectedly.
