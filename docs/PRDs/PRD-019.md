# 1. Executive Summary

The Translator Interface defines the standardized interface between the Claire Performance Engine (CPE) and compatible acoustic backends.

Its responsibility is to receive a valid Performance Representation and translate it into backend-specific instructions.

The Translator Interface performs no communication reasoning and generates no audio.

-------------------------------------------------------------------------------

# 2. Background

The Claire Performance Engine produces a backend-independent Performance Representation.

Compatible acoustic backends require backend-specific instructions.

The Translator Interface separates these concerns while preserving backend independence.

-------------------------------------------------------------------------------

# 3. Objectives

The Translator Interface SHALL:

- Accept a valid Performance Representation.
- Define a common interface for all compatible backends.
- Preserve backend independence within CPE.
- Allow each backend to implement its own translator.
- Remain extensible for future backends.

-------------------------------------------------------------------------------

# 4. Scope

## In Scope

- Translator interface definition.
- Input validation.
- Output contract.
- Backend abstraction.

## Out of Scope

- Backend implementation.
- Audio generation.
- Communication reasoning.
- Acoustic inference.
- Runtime optimization.

-------------------------------------------------------------------------------

# 5. Functional Requirements

The Translator Interface SHALL:

- Accept a valid Performance Representation.
- Reject invalid input.
- Produce backend-specific instructions.
- Remain deterministic.
- Define no backend-specific behavior.

-------------------------------------------------------------------------------

# 6. Interface Architecture

```text
Performance Representation
           │
           ▼
   Translator Interface
           │
           ▼
Backend-specific Translator
           │
           ▼
 Compatible Backend
```

Each backend SHALL implement its own translator using this interface.

-------------------------------------------------------------------------------

# 7. Translator Responsibilities

Every translator SHALL:

- Accept only valid Performance Representations.
- Translate supported information.
- Ignore unsupported information safely.
- Never modify the original Performance Representation.
- Produce deterministic output.

-------------------------------------------------------------------------------

# 8. Design Principles

The Translator Interface SHALL:

- Remain backend-independent.
- Remain implementation-independent.
- Support future compatible backends.
- Preserve immutable input.
- Preserve deterministic behavior.

# 10. Acceptance Criteria

Implementation SHALL:

- Define a reusable translator interface.
- Accept valid Performance Representations.
- Reject invalid input.
- Preserve immutable input.
- Remain backend-independent.
- Support multiple compatible translators.

-------------------------------------------------------------------------------

# 11. Future Work

Future compatible backends SHALL implement translators through this interface.

Examples include:

- Claire Acoustic Model
- Kokoro
- StyleTTS2

Additional compatible backends may be introduced without modifying the interface.

-------------------------------------------------------------------------------

# 12. AI Instructions

When implementing the Translator Interface:

1. Preserve backend independence.
2. Never modify the Performance Representation.
3. Keep backend logic outside CPE.
4. Preserve deterministic behavior.
5. Maintain Constitution compliance.
6. Maintain RFD compliance.
