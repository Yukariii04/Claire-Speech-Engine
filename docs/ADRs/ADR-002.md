# Context

Modern TTS systems generally fall into two categories.

## Category A — Voice Cloning

Examples

- XTTS
- OpenVoice
- CosyVoice
- Fish Speech (voice cloning mode)

Pipeline

```
Reference Audio
      │
Speaker Encoder
      │
Speaker Embedding
      │
Text
      │
Acoustic Model
      │
Audio
```

Advantages

- Supports many speakers
- Little training required
- Voice transfer

Disadvantages

- Higher latency
- Requires reference audio
- Speaker encoder overhead
- Less control over expressive acting
- Harder to create a deeply consistent character

---

## Category B — Dedicated Voice Models

Pipeline

```
Performance Plan
       │
Dedicated Acoustic Model
       │
Dedicated Vocoder
       │
Audio
```

Advantages

- Faster
- Lower VRAM
- Better consistency
- Better emotional control
- Better streaming

Disadvantages

- One model per voice
- Requires dataset collection
- Requires training/fine-tuning

---

# Decision

Claire Speech Engine SHALL adopt the **Dedicated Voice Model Architecture**.

CSE SHALL NOT require reference audio during inference.

Reference audio may be used only during:

- dataset creation
- evaluation
- future fine-tuning

Never as part of normal runtime.

---

# Rationale

Claire is not intended to be

> "any voice."

Claire is intended to become

> one believable character.

Consistency is more valuable than speaker flexibility.

---

# Voice Ownership

The Speech Engine SHALL remain completely independent from any specific voice.

Instead

```
Claire Speech Engine

↓

Voice Package

↓

Audio
```

Voice Packages may contain

- acoustic checkpoints
- vocoder checkpoints
- pronunciation resources
- metadata

The engine itself shall not contain voice-specific logic.

---

# Voice Package Specification

Example

```
voices/

    claire/

        acoustic/

        vocoder/

        metadata.yaml

        pronunciation/

    luna/

    robot/

    narrator/
```

Future PRDs will define the exact package format.

---

# Training Philosophy

CSE SHALL separate

Software

from

Models.

Software includes

- Language Layer
- CIR
- Performance Layer
- Runtime
- Streaming
- APIs

Models include

- Acoustic Model
- Vocoder

These evolve independently.

---

# Initial Development Strategy

Development shall proceed in three phases.

## Phase 1

Implement the complete engine.

No custom model.

A temporary acoustic backend may be used for testing.

---

## Phase 2

Collect a professional speech dataset.

Target

20–40 hours

Single speaker

High quality

Emotionally expressive

---

## Phase 3

Train or fine-tune

Dedicated Claire Voice Package

Replace temporary backend

No architectural changes required.

---

# Why Not Train From Scratch Immediately?

Training a complete acoustic model from zero would require

- enormous compute
- long iteration cycles
- difficult debugging
- architecture research

The project objective is

Speech Engine

not

Speech Research.

Therefore

CSE SHALL first mature its software architecture before investing in model training.

---

# Why Fine-Tuning Is Acceptable

Fine-tuning does not reduce ownership of the engine.

The uniqueness of CSE comes primarily from

- CIR
- Performance Layer
- Prosody Planning
- Timing
- Breathing
- Streaming

The acoustic backend is intentionally replaceable.

---

# Architectural Rules

The engine SHALL never assume

XTTS

OpenVoice

Kokoro

StyleTTS

Fish Speech

or any other backend.

Instead

Every backend SHALL implement the same interface.

Example

```
class AcousticBackend:

    initialize()

    synthesize()

    stream()

    shutdown()
```

Future PRDs will define this interface formally.

---

# Long-Term Goal

Eventually

```
CSE

↓

Performance Plan

↓

Claire Acoustic Model

↓

Claire Vocoder

↓

Audio
```

No external TTS dependency.

No reference audio.

No speaker encoder.

Only the dedicated Claire Voice Package.

---

# Consequences

Future PRDs SHALL be written assuming

- one dedicated voice
- trainable backend
- backend abstraction
- streaming-first inference

No future module shall depend upon voice cloning.
