# Context

Traditional Text-to-Speech systems generally follow this pipeline:

```
Text
 ↓
Normalization
 ↓
Phonemes
 ↓
Acoustic Model
 ↓
Vocoder
 ↓
Audio
```

While this produces intelligible speech, it does not explicitly model **performance**. The acoustic model is expected to infer pauses, emphasis, pacing, hesitation, emotional delivery, and speaking style directly from text or coarse emotion labels.

This architecture makes expressive behavior difficult to control, difficult to debug, and difficult to extend.

Claire Speech Engine is intended to prioritize **believable performance** rather than simply generating natural speech.

---

# Decision

Claire Speech Engine SHALL introduce a dedicated **Performance Layer** between language processing and acoustic synthesis.

The canonical pipeline becomes:

```
Text
    │
    ▼
Language Layer
    │
    ▼
Performance Layer
    │
    ▼
Acoustic Layer
    │
    ▼
Audio
```

The Performance Layer is responsible for deciding **how** the sentence should be performed before any waveform generation begins.

---

# Responsibilities of Each Layer

## Language Layer

Purpose:

Understand the linguistic structure of the input.

Outputs include:

- normalized text
- clauses
- punctuation
- dialogue boundaries
- sentence structure
- speech units

The Language Layer must never decide vocal performance.

---

## Performance Layer

Purpose:

Transform language into an expressive performance plan.

Responsibilities include:

- pause planning
- hesitation planning
- emphasis planning
- speaking rate planning
- breathing placement
- timing
- emotional progression
- intensity modulation
- performance metadata generation

This layer does **not** generate audio.

It produces a structured **Performance Plan**.

---

## Acoustic Layer

Purpose:

Convert the Performance Plan into speech audio.

Responsibilities include:

- phoneme realization
- acoustic synthesis
- vocoder inference
- streaming audio generation

The Acoustic Layer must **never invent** expressive behavior that is absent from the Performance Plan.

---

# Rationale

Separating performance from synthesis provides several benefits:

- expressive behavior becomes deterministic
- easier debugging
- easier testing
- independent replacement of acoustic models
- compatibility with multiple TTS backends
- clearer engineering boundaries

---

# Rejected Alternatives

## Alternative A

Embed all expressive behavior directly into the acoustic model.

Rejected because:

- difficult to debug
- difficult to control
- requires retraining for behavior changes
- tightly couples acting with synthesis

---

## Alternative B

Use discrete emotion labels only.

Rejected because:

- insufficient expressive resolution
- difficult to represent mixed emotions
- encourages static delivery

---

# Consequences

Future PRDs SHALL target the Performance Layer as the primary location for expressive behavior.

Acoustic models are expected to faithfully render the Performance Plan rather than determine it.

---

# Architectural Rules

The Performance Layer SHALL:

- be deterministic
- be independently testable
- expose a stable public API
- remain independent of the acoustic backend

The Acoustic Layer SHALL:

- accept only Performance Plans
- not inspect raw text directly

---

# Future Work

Future PRDs will define:

- Language Representation
- Performance Attributes
- Performance Planner
- Prosody Generator
- Streaming Acoustic Engine

All future components shall comply with this ADR.
